# 오늘 배운 점

파이썬 공식 문서 9장(클래스)을 읽고 정리한다. 지금까지 클래스 문법 자체는 다뤘지만, 그 문법이 내부적으로 어떻게 동작하는지(스코프, 이름 공간, 이름 뒤섞기, 이터레이터 프로토콜)는 새로 알게 된 내용이다.

## 1. 이름 공간과 스코프는 별개의 개념이다

- **이름 공간**은 "이름 → 객체" 매핑이다. 내장 이름 공간, 모듈의 전역 이름 공간, 함수 호출마다 생기는 지역 이름 공간이 각각 따로 존재한다.
- 서로 다른 이름 공간에 있는 이름끼리는 아무 관계가 없다. 그래서 두 모듈이 똑같은 이름의 함수를 각자 정의해도 충돌하지 않는다.
- **스코프**는 이름 공간을 직접 참조할 수 있는 코드 상의 범위를 뜻한다. 실행 중에는 항상 지역 → 감싸는 함수 → 전역 → 내장, 이 순서로 이름을 찾는다.
- 가장 안쪽 스코프에서 `global`이나 `nonlocal` 선언 없이 어떤 이름에 값을 대입하면, 그 이름은 항상 가장 안쪽(지역) 스코프에 새로 생긴다. 바깥 변수를 읽을 수는 있어도, 선언 없이 쓰려고 하면 바깥 변수를 바꾸는 게 아니라 지역에 새 변수가 생긴다는 점이 헷갈리기 쉬운 부분이다.

```python
def outer():
    message = 'outer message'

    def inner():
        nonlocal message
        message = 'changed by inner'

    inner()
    return message

outer()
```

이 코드를 실행하면 `'changed by inner'`가 반환된다. `nonlocal` 없이 `inner` 안에서 `message = ...`를 썼다면, `outer`의 `message`는 그대로 두고 `inner`만의 새 지역 변수가 생겼을 것이다.

## 2. 클래스 객체 · 인스턴스 객체 · 메서드 객체는 서로 다른 개체다

- `class` 블록이 끝까지 정상적으로 실행되면 **클래스 객체**가 만들어진다. 클래스 객체는 어트리뷰트 참조(`클래스.속성`)와 인스턴스 생성(`클래스()`) 두 가지 연산을 지원한다.
- `인스턴스.메서드`로 접근하면, 파이썬은 그 이름을 클래스에서 찾은 뒤 인스턴스와 함수를 하나로 묶은 **메서드 객체**를 만들어 돌려준다. 그래서 `인스턴스.메서드()` 호출은 실제로는 `클래스.메서드(인스턴스)`와 동일하게 동작한다.
- 이 메서드 객체는 변수에 따로 저장해뒀다가 나중에 호출할 수도 있다.

```python
class Timer:
    def tick(self):
        return 'tick'

t = Timer()
savedMethod = t.tick
savedMethod()
```

`savedMethod`에는 `t`와 `tick` 함수가 이미 묶인 메서드 객체가 저장되어 있으므로, `savedMethod()`만 호출해도 `t.tick()`과 똑같이 `'tick'`이 반환된다.

## 3. 클래스 변수는 공유되고, 가변 객체를 클래스 변수로 두면 사고가 난다

- 클래스 변수는 모든 인스턴스가 공유하고, 인스턴스 변수는 인스턴스마다 독립적이다.
- 문제는 리스트나 딕셔너리처럼 값이 바뀔 수 있는(가변) 객체를 클래스 변수로 선언했을 때 생긴다. 인스턴스마다 별도로 가지는 것처럼 보이지만, 실제로는 모든 인스턴스가 똑같은 객체 하나를 같이 쓰고 있어서 한 인스턴스에서 바꾼 내용이 다른 인스턴스에도 나타난다.
- 인스턴스마다 독립적인 리스트나 딕셔너리가 필요하면, 반드시 `__init__` 안에서 `self.속성 = []`처럼 매번 새로 만들어야 한다.

```python
class WrongCart:
    items = []  # 잘못된 예: 모든 인스턴스가 이 리스트 하나를 공유한다

    def add(self, item):
        self.items.append(item)

class RightCart:
    def __init__(self):
        self.items = []  # 올바른 예: 인스턴스마다 새 리스트를 만든다

    def add(self, item):
        self.items.append(item)

wrong1 = WrongCart()
wrong2 = WrongCart()
wrong1.add('apple')
wrong2.add('banana')
wrong1.items  # ['apple', 'banana'] -- 의도치 않게 둘 다 섞인다

right1 = RightCart()
right2 = RightCart()
right1.add('apple')
right2.add('banana')
right1.items  # ['apple'] -- 각자 독립적으로 관리된다
```

## 4. 상속에서 어트리뷰트 검색은 자식에서 시작해서 부모로 올라간다

- 자식 클래스에서 어떤 어트리뷰트를 찾지 못하면 부모 클래스에서 찾고, 부모도 없으면 그 부모의 부모로 계속 올라간다.
- 인스턴스와 클래스 양쪽에 같은 이름의 어트리뷰트가 있으면, 인스턴스 쪽이 항상 우선한다.
- `isinstance(객체, 클래스)`는 어떤 객체가 특정 클래스(또는 그 자식 클래스)의 인스턴스인지 확인한다.
- `issubclass(자식클래스, 부모클래스)`는 클래스 사이의 상속 관계 자체를 확인한다.

```python
class Vehicle:
    pass

class Car(Vehicle):
    pass

myCar = Car()
isinstance(myCar, Vehicle)   # True: Car는 Vehicle을 상속받았다
issubclass(Car, Vehicle)     # True: Car는 Vehicle의 서브 클래스다
issubclass(Vehicle, Car)     # False: 반대 방향은 성립하지 않는다
```

## 5. 다중 상속에서는 MRO(메서드 결정 순서)가 검색 순서를 정한다

- 여러 부모 클래스를 동시에 상속받으면, 파이썬은 "왼쪽에서 오른쪽으로, 같은 클래스는 한 번만" 원칙으로 검색 순서를 정한다.
- 모든 클래스는 결국 `object`까지 이어지기 때문에, 다중 상속 구조는 항상 같은 조상에 여러 경로로 도달하는 모양이 된다. 이 때문에 단순히 순서대로 왼쪽 부모부터 쭉 타고 올라가는 것이 아니라, 어떤 부모도 두 번 검색하지 않도록 순서를 동적으로 정리한 것이 MRO다.
- 실무에서는 다중 상속 구조가 복잡해지기 쉬우므로, 정말 필요한 경우가 아니면 신중하게 사용해야 한다.

## 6. "비공개 변수"는 강제가 아니라 관례와 이름 뒤섞기로 지원된다

- 파이썬에는 클래스 바깥에서 절대 접근할 수 없는 진짜 비공개 변수가 없다. 대신 이름이 밑줄(`_`)로 시작하면 "공개 API가 아니다"라는 관례로 취급한다.
- 밑줄 두 개로 시작하고 끝에는 밑줄이 최대 한 개인 이름(예: `__balance`)은 "이름 뒤섞기"가 적용되어, 실제로는 `_클래스이름__balance` 형태로 저장된다.
- 이 기능의 목적은 완벽한 은닉이 아니라, 자식 클래스가 실수로 같은 이름의 속성을 만들어서 부모 클래스의 내부 로직을 깨뜨리는 사고를 방지하는 것이다.

```python
class Account:
    def __init__(self, balance):
        self.__balance = balance  # 실제 저장 이름은 _Account__balance

    def getBalance(self):
        return self.__balance

acc = Account(1000)
acc.getBalance()          # 1000
acc._Account__balance     # 1000, 뒤섞인 실제 이름으로는 여전히 접근 가능하다
```

## 7. 단순한 데이터 묶음에는 `dataclass`가 어울린다

- 이름 붙은 값 여러 개를 그냥 묶어서 보관하고 싶을 뿐이라면, `__init__`을 직접 작성하는 대신 `dataclasses.dataclass`를 쓰는 것이 더 간결하다.
- `@dataclass`를 클래스 위에 붙이고 속성 이름과 타입을 적으면, `__init__`이 자동으로 만들어진다.

```python
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int

p = Point(3, 5)
p.x, p.y
```

## 8. `for` 문 뒤에는 이터레이터 프로토콜이 숨어 있다

- `for` 문이 실행되면, 파이썬은 내부적으로 대상 객체에 `iter()`를 호출해서 이터레이터를 얻고, 매 반복마다 `next()`로 다음 값을 꺼낸다.
- 더 꺼낼 값이 없으면 이터레이터는 `StopIteration` 예외를 일으키고, `for` 문은 이 예외를 신호로 받아서 자연스럽게 반복을 끝낸다.
- 직접 만든 클래스에 `__iter__`와 `__next__`를 정의하면, 그 클래스의 인스턴스도 `for` 문에 바로 사용할 수 있다.

```python
class CountDown:
    def __init__(self, start):
        self.current = start

    def __iter__(self):
        return self

    def __next__(self):
        if self.current <= 0:
            raise StopIteration
        value = self.current
        self.current -= 1
        return value

for number in CountDown(3):
    print(number)
```

이 코드는 `3, 2, 1`을 차례로 출력한 뒤 `current`가 `0`이 되는 순간 `StopIteration`이 발생해서 반복이 끝난다.

## 9. 제너레이터는 이터레이터를 훨씬 짧게 만드는 방법이다

- `__iter__`와 `__next__`를 직접 구현하는 대신, 함수 안에서 `yield`를 쓰기만 하면 파이썬이 이터레이터 동작을 자동으로 만들어준다.
- `yield`를 만나면 그 자리에서 값을 내보내고 실행을 멈추며, 다음 `next()` 호출 때 멈췄던 지점부터 다시 이어서 실행한다.
- 지역 변수와 실행 위치를 파이썬이 알아서 기억해주기 때문에, `self.current` 같은 인스턴스 변수를 따로 관리할 필요가 없다.

```python
def countDown(start):
    current = start
    while current > 0:
        yield current
        current -= 1

for number in countDown(3):
    print(number)
```

앞서 클래스로 만든 `CountDown`과 똑같이 `3, 2, 1`을 출력하지만, 코드는 훨씬 짧다.

## 10. 제너레이터 표현식은 리스트 컴프리헨션의 괄호 버전이다

- 대괄호 `[]` 대신 소괄호 `()`를 쓰면, 리스트를 통째로 만들지 않고 필요한 값만 그때그때 계산하는 제너레이터 표현식이 된다.
- `sum()`, `max()`, `set()`처럼 값을 하나씩 순서대로 받아서 처리하는 함수에 바로 넘길 때 특히 메모리를 아낄 수 있다.

```python
scores = [88, 92, 79, 95]
sum(score for score in scores if score >= 90)
```

리스트를 먼저 다 만들지 않고, 조건을 만족하는 점수만 하나씩 꺼내면서 바로 합산하므로 `187`이 반환된다.

## 오늘의 한 줄 정리

파이썬 클래스는 "새 문법"이 아니라 기존의 이름 공간·스코프 규칙 위에 살짝 새로운 개념(클래스 객체, 메서드 객체, 이름 뒤섞기)을 얹은 것이다. 그리고 이터레이터·제너레이터는 클래스의 `__iter__`/`__next__` 프로토콜을 이해하고 나면, `for` 문이 왜 그렇게 다양한 객체에 다 동작하는지 이해가 된다.
